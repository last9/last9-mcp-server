Create a static or supported anomaly-macro alert rule in an existing Last9 alert group. Start with recommend_alert_config, then discover and verify the entity, KPI/query, unit, and label dimensions. Pass entity_id separately and the complete typed rule body in alert_rule. Bind primary_indicator and expression_args to real KPIs owned by the entity; do not invent IDs. For static rules, condition, alert_condition, and eval_window are required; eval_window is 0-60 minutes. Clients cannot set algorithm directly—the backend derives it from expression, and currently treats adaptive expressions as static. Put responder context in properties: description, runbook as {"link":"https://..."}, annotations such as owner, service, environment, summary, dashboard, logs, traces, and escalation policy, and query_mode (builder or code). Annotation templates may use {{ $labels.<name> }} and {{ $value }}. Notification routing is configured on the alert group, not the rule. Create new rules with is_disabled=true, validate them, then promote them with patch_alert. Returns the created rule ID and a link to Alerting.
